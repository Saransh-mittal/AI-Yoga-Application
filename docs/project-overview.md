# AI Yoga Breathing Application - In-Depth Project Architecture

> **Authorship Note:** I conceptualized the entire architecture, feature ideation, and the innovative idea to utilize the OpenAI API's new triage agents feature for this application. While the codebase itself was 100% generated by AI based on my directions, all product, UX, and engineering decisions were mine.

## 1. What does the app do?
It's an intelligent, interactive AI Yoga Guide focused on Anulom Vilom (Alternate Nostril Breathing). It gives you a customizable, real-time visual and audio-guided breathing experience.

## 2. Why did I build it?
I was on a break from my startup and mainly I built this primarily as a personal wellness tool to customize my yoga breathing routines and improve my mental health, I even use it daily till this date. But more importantly, while going through the OpenAI documentation, I saw their new agents feature (specifically triage agents) and realized this project would be a perfect, practical sandbox to rigorously test and implement those capabilities myself in a real-world, highly interactive UX environment.

## 3. Tech Stack
*   **Frontend:** Next.js 15, React 19, Tailwind CSS, Framer Motion
*   **Backend:** Next.js API Routes (`app/api/chat`) and a dedicated Python FastAPI backend (`xtts_service`) for handling voice cloning and TTS audio generation.
*   **AI/Agents:** `@openai/agents` framework, OpenAI Models (`gpt-5-nano`, `gpt-5-mini`, `gpt-5`)
*   **Audio Processing:** Client-side Web Audio API (for pristine audio buffer playback and oscillator-based phase transition beeps)
*   **Real-time Communication:** Server-Sent Events (SSE) streaming for tracking background Python tasks.

## 4. Features
*   **Guided Breathing Interface:** Visual representation of Anulom Vilom breathing phases.
*   **Smart Chat Panel:** Interact with an AI assistant to modify routines, update instructions, or configure app settings through chatting only.
*   **Audio Voice Guidance:** Real-time spoken instructions for different phases (inhale, hold, exhale).
*   **Custom Voice Packs:** Upload voice samples to create your own personalized voice instructions using XTTS text-to-speech.
*   **Granular Timers:** Adjustable session duration, inhale duration, hold duration, and exhale duration.
*   **Onboarding Tour:** First-time user setup experience.
*   **Phase Beeps:** Optional synthesized oscillator sounds to mark phase transitions.
*   **Floating Particle Animations:** Calming background visual effects using Framer Motion.

## 5. How is AI Integrated? (The Triage Architecture)
AI is not just bolted on—it constitutes the entire control plane of the application. I designed an Agentic Routing architecture leveraging a sophisticated Factory Pattern:

*   **The Silent Router (Triage Agent):** The primary entry point for all chat messages is the `TriageAgent`, which I enforce to run on `gpt-5-mini` because routing decisions need to be lightning-fast and cost-efficient. My explicit instruction to this router is to be completely invisible—it silently classifies the intent (Settings, Voice Customization, Yoga QA) and immediately hands off execution without outputting any text of its own.
*   **Dynamic Specialized Agents:** When handing off, execution jumps to specialized agents (`SettingsControlAgent`, `VoiceCustomizationAgent`, or `YogaGuideAgent`). Crucially, I implemented a factory pattern so these specialized agents are instantiated dynamically using the *user's preferred model* (e.g., `gpt-5-nano` or `gpt-5`). This means complex, creative tasks like generating new voice scripts are handled by higher-quality models, while routine navigation is kept cheap and fast.
*   **Strict JSON Schemas:** For agents like `VoiceCustomizationAgent` and `SettingsControlAgent`, I enforced strict Zod-based structured schemas. When the user types "make the voice slower," the agent parses this and outputs an exact JSON payload that the frontend intercepts to instantly trigger internal states and backend systems.

## 6. Deep Dive into the XTTS Backend & UX Integration
I wanted a truly seamless, premium UX for voice generation. Generating high-quality audio using the Coqui `xtts_v2` model is computationally expensive and blocking, so I architected an asynchronous, two-step progression system:

*   **The Python XTTS Backend:**
    Built on FastAPI, the backend handles complex multithreading. When a request to update voice instructions hits the API, the endpoint triggers a `BackgroundTasks` execution (`_update_voice_pack_background`).
    To ensure the heavily blocking `tts_to_file` generation function doesn't freeze the FastAPI event loop, I wrapped it inside `asyncio.to_thread`.
*   **Stateful Progress Tracking (`VoicePackManager`):**
    The background threads report continuous progress (e.g., 55% complete generating left-inhale variants) into a central in-memory `_progress_tracker` dictionary via operation IDs.
*   **The Chatbot SSE Integration:**
    Instead of generic loading spinners blocking the entire page, I incorporated the SSE tracking *directly into the AI chatbot*.
    When the backend begins generation, it returns an `operation_id`. The React frontend opens an `EventSource` connection to `/api/voice-packs/progress/{operation_id}`.
    Inside the `ChatPanel.tsx` voice preview block, a native `VoiceProgressBar` renders real-time tracking percentages and human-readable states natively injected over SSE (e.g., "Generating right-exhale audio files... 45%").
    This non-blocking UX allows the user to browse their newly generated scripts while the backend churns through audio rendering in the background.

## 7. Was it deployed or just local?
I built this primarily as a powerful local environment setup. The architecture incorporates a Dockerfile and complex backend microservices (`xtts_service`), encompassing both standard web servers and heavy Python audio-processing systems. It has deployment mechanisms configured for Vercel (Next.js) and Docker, but the full experience (with the XTTS TTS engine) operates as a multi-container local stack.

## 8. Anything unique or cool I feel about it?
*   **Conversational UI for Deep State Management:** Users bypass traditional settings menus entirely. Bouncing between the UI state and natural language operates flawlessly.
*   **Lossless Audio Experience:** I bypass the standard HTML `<audio>` elements. By decoding audio files directly into `AudioBuffers` and streaming through an `AudioContext` `GainNode`, I ensured pristine, zero-latency playback that won't clip.
*   **Seamless Async UX:** Wrapping the heavyweight Coqui XTTS inference inside asyncio threads, managing state in memory, and delivering real-time progress bars straight into a chat bubble results in an incredibly premium application feel.
